🔴 Prioridade 1 — Crítico (necessário para o sistema funcionar end-to-end)
Prioridade Feature Módulo Descrição / O que fazer
P1 — Crítico Integrar Frontend com API Full-Stack Criar camada de serviços HTTP, substituir mock-data.ts, conectar AuthContext ao endpoint /auth/login. Sem isso o projeto não é full-stack.
P1 — Crítico Página "Meus Pedidos" Customer Rota /orders/customer na API já existe. Criar a página frontend que lista os pedidos do usuário logado.
P1 — Crítico Seller pode marcar pedido como Enviado SellerOrders Atualmente a lógica proíbe seller de editar pedidos. Criar endpoint específico para seller marcar SHIPPED seus próprios items.
P1 — Crítico Upload de Imagem do Produto Produto Sem imagem real, o marketplace não tem apelo visual. Adicionar campo imageUrl no schema, endpoint de upload (multer/cloudinary), exibir no frontend.
P1 — Crítico Verificação de Assinatura do Webhook PayPal Payments handleWebhook() aceita qualquer requisição POST. Adicionar verificação do header PAYPAL-TRANSMISSION-SIG para evitar fraudes.
🟡 Prioridade 2 — Alta (necessário para o produto ser utilizável)
Prioridade Feature Módulo Descrição / O que fazer
P2 — Alta Recuperação de Senha Auth Fluxo "esqueci minha senha": geração de token temporário, envio de email, reset via link.
P2 — Alta Perfil do Usuário / Seller Users Página /profile para editar dados. API de PATCH /users/:id já existe. Criar página frontend.
P2 — Alta Rastreamento de Status do Pedido Orders Timeline visual na página do pedido mostrando PENDING → CONFIRMED → SHIPPED → DELIVERED.
P2 — Alta E-mails Transacionais Notifications Confirmação de pedido, aprovação de seller, mudança de status. Usar Nodemailer + Resend/SendGrid.
P2 — Alta Solicitar Saque (Seller) Seller Criar model Withdrawal no schema. Endpoint para seller solicitar saque. Admin processa.
P2 — Alta Sistema de Rank dos Sellers Seller Adicionar campo rank ao model Seller. Cálculo automático baseado em totalSales e rating.
P2 — Alta Rate Limiting no Auth Segurança Adicionar express-rate-limit nos endpoints /auth/login e /auth/register.
P2 — Alta CORS configurado Segurança Substituir cors() aberto por cors({ origin: process.env.ALLOWED_ORIGIN }).
🔵 Prioridade 3 — Média/Baixa (eleva a qualidade do produto)
Prioridade Feature Módulo Descrição
P3 — Média Wishlist / Favoritos Customer Salvar produtos para comprar depois. Novo model no schema.
P3 — Média Perfil Público do Seller Seller Rota /seller/:id pública mostrando loja, produtos e avaliações.
P3 — Média Conversa Comprador ↔ Seller Mensagens Sistema de mensagens assíncrono dentro da plataforma.
P3 — Média Cálculo Automático de Rating Reviews Atualizar seller.rating e product.rating via trigger/query ao criar review.
P3 — Média Analytics Dashboard (Admin) Admin Gráficos de receita, pedidos e usuários ativos. Recharts já está instalado no frontend.
P3 — Média Testes no Backend Qualidade Unit tests nos serviços de auth, orders e payments. Usar Vitest ou Jest.
P3 — Média README Profissional Portfólio Substituir template do Lovable por README real com arquitetura, screenshots e como rodar.
P3 — Baixa Notificações In-App UX Sino de notificações para alertas de pedidos, mensagens e aprovações.
P3 — Baixa Role como Enum no Prisma DB Migrar campo role de String para enum Role { CUSTOMER SELLER ADMIN }.
P3 — Baixa Docker Compose DevOps Arquivo docker-compose.yml para subir o projeto com um comando.
💡 Ordem de Desenvolvimento Recomendada Começar pelo P1 — Integração frontend ↔ API é o passo mais impactante: ele faz o projeto virar full-stack de verdade e desbloqueia testar todas as outras features visualmente. Após isso, o fluxo upload de imagem → email transacional → recuperação de senha torna o produto utilizável por usuários reais. As features de P3 como analytics, chat e wishlist podem ser desenvolvidas em sprints posteriores sem bloquear o core do sistema.
